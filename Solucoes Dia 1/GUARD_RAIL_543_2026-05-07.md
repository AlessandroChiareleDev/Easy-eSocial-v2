# Erro 543 — Causa REAL: colisão de @Id (não era "XML idêntico")

**Data:** 2026-05-07
**Empresa:** Soluções Serviços Terceirizados (CNPJ 09.445.502/0001-09)
**Período afetado:** 2025-08

---

## 0. Mea-culpa do Copilot

A versão anterior deste documento dizia que o erro 543 acontecia porque o
"XML novo era idêntico ao anterior" e que a solução era um guard-rail
comparando o corpo dos XMLs. **Estava errado.** Eu (Copilot) li a palavra
"identificador" na mensagem do eSocial e supus que era "conteúdo" sem
pesquisar a regra. Codei um guard-rail baseado nessa suposição. Ele
funcionou no teste por **coincidência** (porque o pipeline atual está
realmente reenviando o mesmo conteúdo, fato que o usuário já tinha
apontado), mas a causa-raiz que eu documentei estava errada.

O usuário precisou exigir explicitamente "PARA DE SUPOR, PESQUISA" para
eu finalmente abrir a mensagem completa do eSocial e descobrir a causa
verdadeira. Esse documento é a versão corrigida.

## 1. O que o eSocial está reclamando, de verdade

Mensagem completa do erro 543 (extraída do nosso DB):

```
401: Conteudo do evento invalido.
| 543: Ja existe na base de dados do Ambiente Nacional um evento com
        mesmo identificador.
        Identificador: ID1094455020000002026050720224400027.
        Evento Recibo: 1.1.0000000040581453902.
```

**"Identificador" = atributo `@Id` do `<evtPgtos>`.**
Não é o conteúdo do evento. É a string única que a gente coloca no XML:

```xml
<evtPgtos Id="ID1094455020000002026050720224400027">
```

Formato fixo (36 caracteres, exigido pelo leiaute eSocial):

```
ID + tpInsc(1) + CNPJ(14) + AAAAMMDDhhmmss(14) + seq(5)
```

| Pedaço        | Exemplo          | Tamanho |
| ------------- | ---------------- | ------- |
| literal       | `ID`             | 2       |
| `tpInsc`      | `1`              | 1       |
| CNPJ padded   | `09445502000000` | 14      |
| timestamp UTC | `20260507202244` | 14      |
| seq           | `00027`          | 5       |

## 2. A causa-raiz

[Easy-eSocial-v2/backend/app/xml_s1210.py](../Easy-eSocial-v2/backend/app/xml_s1210.py) tinha:

```python
def _gerar_id(tp_insc: int, nr_insc: str, seq: int = 1) -> str:
    now = datetime.now(timezone.utc)
    nr_insc_padded = nr_insc.ljust(14, "0")[:14]
    ts = now.strftime("%Y%m%d%H%M%S")           # precisao de SEGUNDO
    return f"ID{tp_insc}{nr_insc_padded}{ts}{seq:05d}"
```

E o caller passava `seq` como **índice dentro do lote** (1, 2, 3 ... até 40).

**Problema em paralelismo:** dois lotes diferentes rodando no mesmo
segundo, cada um começando o seq do 1, geram exatamente os mesmos
`@Id`s. Exemplo do envio 20 (5 workers paralelos, batch=50):

| Worker A (lote X) | Worker B (lote Y)         |
| ----------------- | ------------------------- |
| `...20224400001`  | `...20224400001` ← colide |
| `...20224400002`  | `...20224400002` ← colide |
| ...               | ...                       |
| `...20224400040`  | `...20224400040` ← colide |

eSocial aceita o `@Id` que chega primeiro (gera recibo `40581...`) e
recusa todos os duplicados subsequentes com 543.

### Confirmação empírica

Script `_invest_543_real.py` testou 10 CPFs com erro 543: para todos os
10, o `@Id` que enviamos foi **byte-a-byte igual** ao `@Id` que o eSocial
diz que já tem. Distribuição dos 117 erros 543:

- envio 20 (paralelo): 109
- envio 21 (paralelo): 8
- demais envios: 0

Bate perfeitamente com a hipótese de colisão entre workers concorrentes.

---

## 3. A correção

[Easy-eSocial-v2/backend/app/xml_s1210.py](../Easy-eSocial-v2/backend/app/xml_s1210.py)
e [Easy-eSocial-v2/backend/app/xml_s1298.py](../Easy-eSocial-v2/backend/app/xml_s1298.py):

```python
_ID_LOCK = threading.Lock()
_ID_LAST_TS: str = ""
_ID_COUNTER: int = 0

def _gerar_id(tp_insc: int, nr_insc: str, seq: int = 1) -> str:
    """`seq` arg eh ignorado. Usamos contador atomico GLOBAL por segundo."""
    global _ID_LAST_TS, _ID_COUNTER
    now = datetime.now(timezone.utc)
    nr_insc_padded = nr_insc.ljust(14, "0")[:14]
    ts = now.strftime("%Y%m%d%H%M%S")
    with _ID_LOCK:
        if ts != _ID_LAST_TS:
            _ID_LAST_TS = ts
            _ID_COUNTER = 0
        _ID_COUNTER += 1
        seq_global = _ID_COUNTER
    return f"ID{tp_insc}{nr_insc_padded}{ts}{seq_global:05d}"
```

Com isso, todos os workers/lotes compartilham o mesmo contador.
Limite teórico: 99.999 IDs por segundo, mais que suficiente.

### Validação

Cenário: 5 threads × 40 IDs cada, no mesmo segundo:

| Antes do fix     | Depois do fix   |
| ---------------- | --------------- |
| 200 IDs gerados  | 200 IDs gerados |
| **40 únicos**    | **200 únicos**  |
| **160 colisões** | **0 colisões**  |

### Por que 459 é o difícil

O recibo guardado em `s1210_cpf_recibo` ficou desatualizado porque
**alguém de fora** (sistema antigo do cliente, contador, etc.) retificou
aquele S-1210 no eSocial. Quando tentamos retificar com o recibo velho,
eSocial responde "esse recibo não está mais ativo".

A única forma de detectar isso antes do envio seria reconsultar o eSocial
**a cada CPF** para pegar o recibo atual — mas isso bate na cota de 10
consultas/dia do Download Cirúrgico. Inviável por CPF.

**Plano para 459:** rodar `ConsultarIdentificadoresEmpregador` (1 call
para todos os eventos do CNPJ no mês) quando a cota estiver liberada,
sincronizar `s1210_cpf_recibo`, depois reenviar.
